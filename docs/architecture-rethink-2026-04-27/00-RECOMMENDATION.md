# Architecture Recommendation: Stop the FOUC, Don't Burn It Down

**Date:** 2026-04-27
**Status:** Decision document
**TL;DR:** **Patch in place this week. Defer Astro to Q3.** The FOUC is fixable in 2-4 hours. Migrating to Astro is a 1-2 week rabbit hole that puts 30+ PRs of recent SEO work and live affiliate revenue at risk for marginal gains your users won't notice.

---

## Why are we running React for what's mostly static content?

Honest answer: **path dependence.** The previous architect optimized for:

1. **shadcn/Radix UI ecosystem** — `package.json` shows 24 `@radix-ui/*` packages, full Tailwind v4, framer-motion. The tooling is real and the components are polished.
2. **Genuine interactivity that exists** — DosageCalculator (multi-input form with live computation), ComparisonWidget, ImageLightbox, scrolling TOC with active-section highlighting, mobile TOC accordion (G1 audit confirms 5 truly interactive surfaces per blog post).
3. **Developer ergonomics** — One mental model (React everywhere), shared `<Layout>`, shared analytics hooks (`useAffiliateTracking`, `useScrollTracking`, `useElementTracking`, `useEngagementTracking`), one router.
4. **JSON-CMS workflow** — 189 posts in `src/newblog/data/posts/*.json`, registry generated by script. Adding a post = drop a JSON file. That's a real workflow asset.

**Was it the right choice?** For a content site, no — G1 confirms ~20% of the page surface is interactive, ~80% is static. The tax is a 412KB main bundle, a 150KB content chunk, render-blocking 177KB Tailwind CSS, and the FOUC you just experienced.

**Would I pick React SPA today for a fresh affiliate content site?** No — I'd pick Astro with React islands. Same components, 70-90% less JS, no FOUC by design, identical DX.

**But you're not on a fresh project.** You have 189 posts, 30+ recent SEO PRs, working Vercel deploy, live revenue, custom prerender, and a JSON-CMS workflow that works.

## The recommendation: Option 1 — Patch in place

**Pick this. Ship Monday.**

### What's actually wrong (confirmed by direct inspection)

The FOUC has two compounding causes:

1. **Style mismatch (G2 confirmed):** `scripts/prerender-blog-posts-enhanced.js:327-341` emits raw `<article><h1>...` with only `class="meta"` and `class="excerpt"`. NewBlogPost.jsx renders `<motion.article className="bg-white rounded-xl shadow-lg ...">` wrapping `<div className="prose prose-lg prose-green ...">`. Browser paints unstyled Times New Roman until JS replaces it.
2. **`createRoot` not `hydrateRoot` (`src/main.jsx:139`):** React calls `createRoot(container)` then `root.render(...)` — this **wipes** the prerendered DOM and remounts from scratch instead of hydrating in place. Even with matching markup, this would still cause a flash.

### The fix (2-4 hours, near-zero risk)

**Change A — Wrap prerender output in the same Tailwind classes as React renders** (`scripts/prerender-blog-posts-enhanced.js:327`):

```js
rootDiv.innerHTML = `
  <div id="prerender-content">
    <div class="min-h-screen bg-gray-50">
      <article class="bg-white rounded-xl shadow-lg overflow-hidden max-w-3xl mx-auto my-8">
        <div class="p-8 md:p-12">
          <h1 class="text-4xl md:text-5xl font-bold text-gray-900 mb-4">${safeTitle}</h1>
          <div class="meta flex gap-4 text-sm text-gray-500 mb-6">...</div>
          <div class="prose prose-lg prose-green max-w-none">
            ${quickAnswerHtml}
            ${fullContentHtml}
          </div>
        </div>
      </article>
    </div>
  </div>
`;
```

The Tailwind classes are already in the 177KB CSS bundle. They'll apply the moment CSS loads — long before JS executes.

**Change B — Inline critical CSS for above-fold styling** (`index.html` head, append to existing `critical-lcp.css` block). 5-10 KB of `prose`, `bg-white`, `rounded-xl`, base typography. Renders styled in the first paint, not after CSS round-trip.

**Change C (optional but cheap) — Switch to `hydrateRoot`** (`src/main.jsx:139`). Replace `createRoot(container)` + `root.render(...)` with `hydrateRoot(container, <App/>)`. Caveat: prerender markup must structurally match React output or you get hydration warnings. **Skip this if matching is too brittle** — Change A alone gets you 90% of the win without hydration-mismatch risk.

### Why this beats hydrate-properly and Astro

- **Patch (Option 1):** 2-4 hr. FOUC drops from 2s of unstyled-Times-New-Roman to ~200ms of styled HTML before React boots. Zero risk to SEO. Zero risk to revenue.
- **Hydrate properly (Option 2):** 1-2 days. Need `renderToString` of NewBlogPost.jsx, which means importing the full React tree (framer-motion, Radix, ReactMarkdown, 24 hooks) into a Node build context. Hydration mismatches across 189 posts = whack-a-mole. G2 estimates 8-16hr and that's optimistic. Marginal user-perceived improvement over Option 1.
- **Astro (Option 3):** 1-2 weeks of rewriting routing, components, prerender, registry, custom hooks. Best long-term answer. But: risks the 30+ recent SEO PRs (canonicals, structured data, redirects, breadcrumbs), the FAQ/Review/HowTo schema generators, custom prerender features (Quick Answer callout, breadcrumb schema, dateModified from git). And you'd be migrating during peak revenue. **Not Monday's problem.**
